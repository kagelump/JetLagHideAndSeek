## New findings

### N1 (blocker) — Bundles were not regenerated after the fixes

The route-by-route comparison of `assets/transit/*.json` against HEAD gives
**byte-for-byte the same result as round 1**: 165 changed, 152 improved, 12
worsened — including のぞみ (`osm:relation:9802526`, 355 → 384 km max gap)
and JR東海道本線 (`osm:relation:11680904`, 389 → 398 km) in 6 bundles each.
Under the fixed code these variants revert atomically, so their geometry
should be _identical to HEAD_ after regeneration; it isn't. The committed-to-be
artifacts (and `data/transit/report/japan.md`, which still shows the inflated
`Repaired stops: 1303`) were generated by the round-1 code.

**Action**: re-run `pnpm data:transit -- --cache-only`, then re-verify:

- のぞみ / 東海道本線 geometry identical to HEAD (atomic revert),
- the ~152 improved routes retained,
- `Repaired stops` in the report drops to the true reinsertion count,
- `Unrepairable variants` rises (partial repairs now count as unrepairable).

### N2 (blocker) — New test suite runs in no test runner

`package.json`'s `test:data:transit` and `pretest` both enumerate test files
explicitly, and **neither includes `stopOrderRepair.test.mjs`**. Both rounds'
"97 passing" came entirely from pre-existing suites; the new tests have never
run in `pnpm test` or `pnpm test:data:transit`. This is what allowed N3 to go
unnoticed.

**Action**: add the file to both scripts (note `pretest` also omits
`osmRoutes.test.mjs` while `test:data:transit` has it — consider a
`data/transit/scripts/lib/*.test.mjs` glob to end this class of bug; that
also picks up the similarly-orphaned `normalizeOperator.test.mjs`).

### N3 (major) — 2 of 11 new tests fail when run directly

`node --test data/transit/scripts/lib/stopOrderRepair.test.mjs` → 9 pass,
2 fail. Both are fixture-calibration bugs, not code bugs:

- **"declines repair for a reversed mid-route block (R5)"** — the fixture's
  trunk spacing is 10 km, so the reversed block's boundary gaps are 30 km
  while the threshold is `max(20, 4 × 20) = 80 km` (median-of-rest is 20 km).
  Detection never fires, `warnings` is empty, the assertion fails. The
  fixture needs tight trunk spacing (~1–2 km) with a block displaced > 20 km
  so the boundary gaps clear both threshold terms. (This failure documents a
  real detector property worth a code comment: a reversed block whose
  boundary gaps stay under 4× median is intentionally undetectable.)
- **"repairs duplicate consecutive ids … (R6)"** — the entire fixture spans
  ~1 km, far below the 20 km absolute floor, so nothing is ever flagged and
  `repaired` is `false`. Needs a long line (reuse the 30-stop ~1 km-spacing
  pattern) containing a duplicated id pair plus a genuinely distant outlier.

Until these pass, the R5 decline/revert behavior and the R6 duplicate-id
handling are unpinned by tests.

### N4 (minor) — New config validation paths lack negative tests

`config.mjs` now rejects null relation overrides and non-`osm:node:<id>`
`stopOrder` elements, but `config.test.mjs` only exercises the round-1 cases.
Add one test each for `"123": null` and `stopOrder: [42]`.
