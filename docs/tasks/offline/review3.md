## The GitHub Pages question: four uses, one site

### What Pages is being asked to serve

1. **Splash/home page** ‚Äî `site/index.html` (+ `privacy.html`, assets)
2. **App deep linking** ‚Äî `site/.well-known/apple-app-site-association` +
   `assetlinks.json`, generated by `scripts/build-share-site-config.mjs`
   (plus the `/i/` import-link fallback page)
3. **Bundle viewer** ‚Äî built by `tools/data-viewer/build.mjs` into
   `site/bundle-viewer/`
4. **Packs catalog** ‚Äî `catalog.json` + `NOTICE` (this epic)

### How they're deployed today ‚Äî two colliding mechanisms

- Uses 1‚Äì3 deploy via **[pages.yml](.github/workflows/pages.yml)**: on
  master pushes touching `site/**` etc., it builds the verification files +
  viewer and ships the `site/` directory through
  `actions/upload-pages-artifact` + `actions/deploy-pages`. Its header
  comment is explicit: _"GitHub Pages must be enabled with GitHub Actions
  as the source."_
- Use 4 deploys via **`publish.mjs` step 5**: commit `catalog.json` +
  `NOTICE` (+ `index.html`) to an orphan **`gh-pages` branch** and push.
  T4/design.md assumed _"the app has no other Pages use"_ ‚Äî that premise
  was already false when the epic was written; `pages.yml` predates it.

GitHub Pages has **exactly one source per repo**: either "GitHub Actions"
or "deploy from a branch". The two mechanisms cannot both be live.

### What is actually live (probed 2026-06-12)

| URL                                        | Status  |
| ------------------------------------------ | ------- |
| `https://jetlag.hinoka.org/` (splash)      | **200** |
| `‚Ä¶/.well-known/apple-app-site-association` | **200** |
| `‚Ä¶/bundle-viewer/`                         | **200** |
| `‚Ä¶/i/`                                     | **200** |
| `‚Ä¶/catalog.json`                           | **404** |
| `‚Ä¶/NOTICE`                                 | **404** |

Pages is (correctly) still on the **Actions** source, so the `gh-pages`
branch ‚Äî which now contains a real two-pack catalog ‚Äî is inert. The app's
`OFFLINE.catalogUrl` can never resolve. Flipping the source to the branch
would invert the failure: catalog up, but splash, deep links (universal
links/app links would break on devices), `/i` fallback, and the viewer all
down ‚Äî strictly worse. **So no: the current setup cannot fulfill all four
goals under either setting of the Pages source.**

Three additional latent conflicts in the branch flow, even beyond the
source setting:

- `publish.mjs` writes a pack-table **`index.html` at the site root**
  ([publish.mjs:429-456](data/packs/scripts/publish.mjs)) ‚Äî if the four
  uses are ever unified, that file collides with the splash page. (The
  current `gh-pages` branch happens not to contain it ‚Äî the branch was
  hand-tended ‚Äî but the next `publish` run will re-add it.)
- The published catalog's `attributionUrl` is the stale flipped domain
  `hinoka.jetlag.org/NOTICE` (published before `ddd6bd6`; both the
  gh-pages copy and `dist/catalog.json` carry it). Needs a republish after
  the strategy lands.
- **[packs-catalog.yml](.github/workflows/packs-catalog.yml) can never
  trigger as designed**: push-triggered workflows are read from the
  _pushed ref_, and the orphan `gh-pages` branch contains no
  `.github/workflows/`. The C7 dual-checkout fix made the job's steps
  correct but the trigger unreachable. (It also doesn't exist on
  `origin/master` yet ‚Äî see P3.)

### Recommendation: one site, one deploy path

Fold the catalog into the existing Actions deployment and retire the
`gh-pages` branch entirely:

1. `publish.mjs` step 5 becomes: write `catalog.json`, `NOTICE`, and the
   pack-table page into **`site/packs/`** on master (namespaced ‚Äî no root
   collisions: `site/packs/catalog.json`, `site/packs/NOTICE`,
   `site/packs/index.html`) and commit+push master. `pages.yml` already
   triggers on `site/**` and deploys atomically with the other three uses.
2. Update `OFFLINE.catalogUrl` ‚Üí
   `https://jetlag.hinoka.org/packs/catalog.json` and the catalog
   `attributionUrl` ‚Üí `‚Ä¶/packs/NOTICE`. (One shared constant for the Pages
   origin, per review 2 ¬ß5, so app/pipeline can't drift again.)
3. Replace `packs-catalog.yml` with a validation step **inside
   `pages.yml`** (or a master-push workflow on `site/packs/catalog.json`):
   run `validateCatalog()` + the HEAD checks _before_ `deploy-pages`, so a
   bad catalog blocks the deploy instead of auditing it after the fact.
   This also fixes the unreachable-trigger problem for free.
4. Delete the gh-pages branch and the worktree/orphan logic from
   `publish.mjs` (a sizeable chunk of its riskiest code), and update
   design.md/T4's "claims the Pages site" paragraph, which is wrong.

Why this direction and not branch-unification: the `.well-known` files
need repo secrets (team id, cert fingerprints) at build time ‚Äî they
_must_ come from CI, not from a dev-machine publish; and deep links are
the one use that hard-breaks user-visible functionality if it ever goes
down. Keep the mechanism that already serves them, move the small new
thing into it. Catalog updates do gain a "commit to master" step ‚Äî for a
solo repo publishing from the dev machine, that's the same `git push`
publish.mjs already does, minus the orphan-branch machinery.

## Other findings (round 3)

### P3. The epic has never been pushed ‚Äî `origin/master` is 5 commits behind

`origin/master` is still at `645fa93` (pre-epic). Consequences: no CI
(app-checks, Maestro) has ever run against the epic; `packs-catalog.yml`
doesn't exist on the remote default branch; the published release and
gh-pages catalog reference code that isn't on the remote. Push (or open
the PR) before anything else ‚Äî a green local run isn't a substitute for
the repo's own CI, and the epic docs' house rules assume PR-per-task
review.

### P4. Taiwan's admin-4th maps to a level with 3 relations

`asia-taiwan` still uses the default `matching: [4, 7, 9, 10]`. The built
artifact has per-level counts **4‚Üí21, 7‚Üí169, 8‚Üí199, 9‚Üí7762, 10‚Üí3** ‚Äî so
`admin-4th` (level 10) has three candidates in the whole country, while
level 8 (urban districts ÂçÄ) is unused. Taiwan's mapping needs curation
(likely `[4, 7, 8, 9]` ‚Äî districts at 8, villages at 9). The N4 lint
warning only fires at _zero_ rows; consider warning below a small
threshold (e.g. <10) or, better, printing the per-level count table for
the `matching` levels on every build so curation is a glance, not a dig.
M2 explicitly uses Taipei admin matching as its exit criterion, so this
matters now, not later.

### P5. Transit data quality (carried, unchanged)

N3 fixed the schema, not the content: NL still has 45 presets with
`other` holding 1354 of 1977 stations (sparse `operator` tags), tram
stops mixed in, heritage one-offs as presets, `hsl()` colors (bundled
data uses hex ‚Äî verify MapLibre iOS `to-color` accepts hsl strings before
shipping), and no reuse of `data/transit/`'s operator
normalization/clustering. Fine to defer per the agreed ordering, but it
remains the gap between "presets load" and "presets are usable" for M3.

### P6. Smaller carried items, quick status
